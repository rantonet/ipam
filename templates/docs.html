Panoramica

IPAM (IP Address Management) è un'applicazione web sviluppata in Python 3 / Flask per la gestione degli spazi di indirizzamento IP, delle subnet e delle VLAN aziendali. In produzione gira su Gunicorn (porta 8000) dietro Apache 2.4 come reverse proxy nella directory dedicata /ipam.

L'URL di accesso è http://<server>/ipam/. Il prefisso viene gestito dal middleware ReverseProxied in wsgi_gunicorn.py: tutti i link interni generati da url_for() includono già /ipam.

Architettura

Stack Tecnologico

  • Backend: Python 3 · Flask · Flask-SQLAlchemy · Flask-Login · Werkzeug
  • Web server: Apache 2.4 (reverse proxy) → Gunicorn :8000 (WSGI)
  • Database: SQLite — file instance/ipam.db
  • Frontend: HTML5 · CSS3 separato · JavaScript vanilla
  • Font: IBM Plex Mono + IBM Plex Sans
  • Dipendenze principali: flask, flask-sqlalchemy, flask-login, werkzeug, gunicorn, dnspython, apscheduler, ldap3

Installazione

L'applicazione gira su Debian · Ubuntu · RHEL · Rocky Linux · AlmaLinux · CentOS · Fedora. Lo script install.sh rileva automaticamente la famiglia della distribuzione, installa le dipendenze, crea il servizio systemd per Gunicorn e configura Apache come reverse proxy.

Architettura di rete: Browser → Apache :80 → ProxyPass → Gunicorn :8000 → Flask

File ipam.conf

Generato automaticamente da install.sh nella directory corretta per la distro:

  • Debian / Ubuntu: /etc/apache2/conf-available/ipam.conf
  • RHEL / Rocky / Alma / CentOS / Fedora: /etc/httpd/conf.d/ipam.conf
# File statici serviti direttamente da Apache (più veloce)
Alias /ipam/static /var/www/html/ipam/static
<Directory /var/www/html/ipam/static>
    Options -Indexes
    AllowOverride None
    Require all granted
</Directory>

# Proteggi il database SQLite
<Directory /var/www/html/ipam/instance>
    Require all denied
</Directory>

# Reverse proxy verso Gunicorn
ProxyPass        /ipam/static !
ProxyPass        /ipam  http://127.0.0.1:8000/ipam
ProxyPassReverse /ipam  http://127.0.0.1:8000/ipam

Aggiungere nel VirtualHost (o in httpd.conf) per il corretto forwarding degli header: 

ProxyPreserveHost On
RequestHeader set X-Forwarded-Host %{HTTP_HOST}s
RequestHeader set X-Forwarded-Proto %{REQUEST_SCHEME}s

File di servizio systemd

Il servizio usa Gunicorn installato nel virtualenv dedicato (venv/) creato dallo script install.sh nella directory dell'applicazione. 

[Unit]
Description=IPAM Gunicorn service
After=network.target

[Service]
User=www-data
WorkingDirectory=/var/www/html/ipam
ExecStart=/var/www/html/ipam/venv/bin/gunicorn --bind 127.0.0.1:8000 --workers 1 --timeout 120 wsgi_gunicorn:application
Restart=always

[Install]
WantedBy=multi-user.target
sudo systemctl restart ipam   # riavvia dopo ogni aggiornamento

Passi di installazione

  1. Installazione automatica (consigliata) — lo script rileva da solo la famiglia della distro, crea un virtualenv Python in /var/www/html/ipam/venv/, installa le dipendenze da requirements.txt, crea il servizio Gunicorn e configura Apache: 
    sudo chmod +x /sorgente/ipam/install.sh
sudo /sorgente/ipam/install.sh
  2. Aggiornamento — usa update.sh che preserva il database e aggiorna le dipendenze Python nel venv: 
    sudo chmod +x /sorgente/ipam/update.sh
sudo /sorgente/ipam/update.sh

Dopo l'installazione l'applicazione è raggiungibile su http://<IP-server>/ipam/. Le credenziali predefinite sono admin / admin (cambiarle subito dalla pagina Impostazioni).

SELinux — nota importante

Su RHEL e derivate SELinux è in modalità enforcing per default. Senza i contesti corretti Apache non riesce a leggere i file Python né a scrivere il database SQLite. I due contesti necessari sono:

  • httpd_sys_content_t — lettura dei file sorgente, template, CSS
  • httpd_sys_rw_content_t — lettura e scrittura della directory instance/ (richiesto da SQLite)

Lo script install.sh applica entrambi automaticamente tramite semanage + restorecon.

Come funziona il prefisso /ipam

Apache trasferisce le richieste a Gunicorn con header X-Forwarded-Host. Il middleware ReverseProxied in wsgi_gunicorn.py ripristina l'hostname originale e imposta SCRIPT_NAME=/ipam. Flask legge SCRIPT_NAME e lo include automaticamente in ogni URL generato da url_for(), quindi non è necessario hardcodare /ipam in nessun template.

Virtualenv Python

L'applicazione usa un virtualenv Python dedicato (venv/) nella directory di installazione, in modo da isolare le dipendenze dal sistema operativo e rendere l'aggiornamento dei pacchetti sicuro e riproducibile.

Creare il virtualenv manualmente

Lo script install.sh crea e popola il venv automaticamente. Se si rende necessario ricrearlo da zero (es. cambio di versione Python, venv corrotto): 

cd /var/www/html/ipam

# Rimuovere il venv esistente (opzionale ma consigliato se corrotto)
sudo rm -rf venv

# Creare un nuovo venv con il Python di sistema
sudo python3 -m venv venv

# Installare tutte le dipendenze
sudo venv/bin/pip install --upgrade pip
sudo venv/bin/pip install -r requirements.txt

# Correggere i permessi (www-data deve poter leggere il venv)
sudo chown -R root:www-data venv/
sudo chmod -R 750 venv/

Installare o aggiornare le dipendenze

Ogni volta che requirements.txt viene modificato (nuova dipendenza o aggiornamento di versione), è necessario sincronizzare il venv: 

cd /var/www/html/ipam
sudo venv/bin/pip install -r requirements.txt
sudo systemctl restart ipam

Per aggiornare tutti i pacchetti all'ultima versione compatibile con i vincoli dichiarati: 

sudo venv/bin/pip install --upgrade -r requirements.txt
sudo systemctl restart ipam

Verificare i pacchetti installati

Per controllare cosa è installato nel venv e le versioni effettive: 

sudo venv/bin/pip list

Per confrontare i pacchetti installati con quelli dichiarati in requirements.txt e individuare dipendenze mancanti o obsolete: 

sudo venv/bin/pip install --dry-run -r requirements.txt

Per vedere quale versione di Python usa il venv: 

venv/bin/python --version

Quando aggiornare il venv

  • Dopo ogni pull / deploy che modifica requirements.txt
  • Quando il servizio si avvia ma produce errori ModuleNotFoundError o ImportError
  • Dopo aver eseguito update.sh (lo script aggiorna già il venv in automatico)
  • Quando si cambia la versione di Python sul server (il venv va ricreato da zero)

Nota: il servizio systemd ipam punta direttamente all'eseguibile venv/bin/gunicorn. Dopo qualsiasi modifica al venv è sempre necessario riavviare il servizio con sudo systemctl restart ipam.

Modello Dati

Network

  • address / cidr — es. 10.122.0.0 / 19
  • name — nome descrittivo
  • network_type — supernet | subnet
  • parent_id — FK ricorsiva per gerarchia supernet → subnet
  • status — active | reserved | deprecated
  • location, country, site, usage, zone, gateway — metadati opzionali
  • vlan_id — FK verso vlans

Proprietà calcolate: total_hosts, used_count, available_count, usage_percent, broadcast, netmask.

IPRecord

  • ip_address — stringa univoca (es. 10.0.0.1)
  • ip_int — rappresentazione intera a 32 bit dell'IPv4 — usata per l'ordinamento O(log n) tramite indice B-tree
  • hostname — nome host risolto via DNS PTR
  • mac_address — indirizzo MAC
  • switch_name — nome switch rilevato via SNMP
  • switch_port — numero porta switch
  • snmp_updated_at — timestamp ultimo aggiornamento SNMP
  • device_type — server | workstation | router | switch | printer | ap | phone | other
  • os_type — sistema operativo (testo libero)
  • status — used | reserved | dhcp | available
  • network_id — FK verso networks
  • description — note libere
  • last_seen — timestamp ultimo rilevamento attivo

Indici DB: ip_int, network_id, network_id+status, status, device_type, switch_name, os_type, last_seen. Tutti creati automaticamente all'avvio da _ensure_indexes().

VLan

  • vlan_id — 1–4094, univoco
  • name — nome descrittivo
  • status — active | deprecated

LocalUser

  • username — univoco
  • password_hash — hash bcrypt
  • is_admin — booleano (almeno un admin deve sempre esistere)

AppConfig

Tabella chiave-valore per la configurazione persistente.

  • dns_primary, dns_secondary — server DNS per PTR lookup
  • global_scan_enabled, global_scan_time — scansione globale schedulata
  • ldap_* — configurazione LDAP / Active Directory
  • snmp_community, snmp_routers — configurazione SNMP

ScanGroup / ScanGroupItem

Gruppi di scansione pianificata con orario dedicato.

  • group_type — supernet | subnet | ip_range
  • schedule_time — orario HH:MM di esecuzione giornaliera
  • Ogni item punta a una network_id o a un ip_range CIDR testuale

ScanLog

Registro automatico di ogni scansione e SNMP discovery.

  • scan_type — subnet | snmp | group
  • target — subnet CIDR o IP router oggetto della scansione
  • started_at, ended_at, duration — timing
  • found, updated, errors — contatori risultati
  • status — running | done | error

Pagine

Scanner

Il modulo scanner.py esegue per ogni IP della subnet:

  1. Ping — verifica se l'host risponde (comando di sistema, N thread paralleli)
  2. DNS PTR lookup — query reverse tramite dnspython verso i server configurati. Se il primario risponde NXDOMAIN, viene interrogato il secondario.
  3. ARP / SNMP — lettura opzionale delle ARP table dai router (configurare SNMP_ROUTERS)

I server DNS vengono letti dalla tabella app_config all'avvio e aggiornati a runtime ad ogni salvataggio nella pagina Impostazioni.

Parametri configurabili (in scanner.py)

  • PING_TIMEOUT — secondi per ping (default: 1)
  • DNS_TIMEOUT — secondi per query DNS (default: 2)
  • MAX_WORKERS — thread paralleli per subnet (default: 50)
  • MAX_SUBNET_WORKERS — subnet in parallelo per scan globale (default: 4)
  • SNMP_COMMUNITY — community SNMPv2c (default: cmfpub)
  • SNMP_ROUTERS — lista IP router da cui leggere le ARP table

Utilizzo da riga di comando

python3 scanner.py --subnet-id 42
python3 scanner.py --subnet 10.114.1.0 --cidr 24
python3 scanner.py --all
python3 scanner.py --all --skip-recent 6    # salta subnet scansionate nelle ultime 6 ore

Scheduler

L'applicazione integra APScheduler (BackgroundScheduler) per l'esecuzione automatica delle scansioni agli orari configurati nella pagina Impostazioni.

Funzionamento con Gunicorn multi-worker

Gunicorn può avviare più worker. Per evitare scheduler duplicati, l'applicazione usa un file lock esclusivo (instance/scheduler.lock via fcntl.LOCK_EX | LOCK_NB). Solo il primo worker che ottiene il lock avvia lo scheduler; gli altri lo ignorano silenziosamente.

Tipi di job schedulati

  • Scansione globale — se abilitata, scansiona tutte le subnet attive all'orario configurato
  • Gruppi di scansione — ogni gruppo abilitato con un orario viene schedulato con CronTrigger

Lo scheduler viene ricaricato automaticamente ad ogni modifica alle impostazioni o ai gruppi (_reload_scheduler_jobs()).

Dipendenza da installare

pip install "apscheduler>=3.6,<4.0"

Script Utility

fill_missing_ips.py

Popola le subnet con tutti gli IP mancanti impostando status='available'. Utile dopo l'importazione di reti da SolarWinds o altri sistemi che non includono la lista completa degli host.

  • Salta reti con CIDR < --cidr-min (default: 17, cioè /16 e più grandi)
  • Salta subnet che hanno subnet figlie (solo le foglie vengono popolate)
  • Usa INSERT OR IGNORE — è idempotente, può essere rieseguito senza danni
python3 fill_missing_ips.py --dry-run          # mostra quanti IP verrebbero aggiunti
python3 fill_missing_ips.py                    # esegue l'inserimento
python3 fill_missing_ips.py --cidr-min 20      # salta reti /19 e più grandi

update.sh

Aggiorna i file dell'applicazione preservando il database e le configurazioni esistenti. Copia i file sorgente, installa le dipendenze aggiornate e riavvia il servizio Gunicorn. 

sudo /sorgente/ipam/update.sh
